# Story 7.1: Création d'un Élève (Données Personnelles)

## Status: Ready for Review

## Story

**As an** Admin/Directeur,
**I want** créer un élève avec ses données personnelles (Étape 1 du parcours d'inscription),
**so that** je puisse ensuite l'associer à un parent/tuteur (Étape 2) et l'inscrire dans une classe (Étape 3).

## Acceptance Criteria

Source: `docs/prd/module-inscriptions.md#Story 1.1 : Creation d'un Eleve (Donnees Personnelles)`

1. Un formulaire "Inscription Élève – Étape 1" expose les champs suivants :
   - Nom (texte, obligatoire)
   - Prénom (texte, obligatoire)
   - Date de naissance (date, obligatoire, antérieure à `today`)
   - Lieu de naissance (texte, optionnel)
   - Sexe (radio : `M`, `F`)
   - Nationalité (select, défaut : `Nigérienne`)
   - Téléphone (texte, optionnel)
   - Adresse (textarea, optionnelle)
   - Ville (texte, défaut : `Niamey`)
   - Quartier (texte, optionnel)
   - Groupe sanguin (select, optionnel)
   - Maladie ou allergie connue (textarea, optionnel)
2. Si un élève avec les mêmes `firstname` + `lastname` + `birthdate` existe déjà (non soft-deleted), l'API retourne un avertissement (sans bloquer) incluant le matricule du doublon : « Un élève avec les mêmes nom, prénom et date de naissance existe déjà (Matricule : XXX). Voulez-vous continuer ? ». Le client peut soit confirmer la création (`force=true`), soit annuler.
3. L'admin peut joindre une photo (formats : `jpg`, `jpeg`, `png`, taille ≤ 2 Mo). En cas de violation : 422 `photo` avec message clair.
4. Le payload accepté retourne le student créé (status 201) avec sa photo persistée (chemin relatif) et le client peut prévisualiser cette photo via l'URL renvoyée (Story 7.4 ajoutera drag&drop + redimensionnement auto — pas requis ici).
5. La création expose un identifiant `student.id` consommable par l'Étape 2 (parent/tuteur). Le matricule **n'est pas encore généré** ici (Story 7.2). À ce stade, `matricule` est NULL en base ou bien réservé via le placeholder `DRAFT-{id}` (cf. Dev Notes).
6. Une annulation (pas d'appel POST) n'écrit rien en base.

### Coding Standards

- `protected $connection = 'tenant';` obligatoire sur tous les models.
- Laravel 12 : `casts()` method, **pas** `$casts` property.
- Form Requests obligatoires — aucune validation inline.
- API Resources obligatoires — aucun retour de model brut.
- Eager loading dès qu'une relation est traversée.
- Réponse JSON standard : `{ message, data }` pour POST/PUT, paginée pour GET index.
- Type hints stricts (PHP 8.3) + return types explicites.
- Pas d'`env()` hors `config/` (utiliser `config('filesystems.default')` etc.).

### Security & Upload

- Photo stockée sur disque `tenant` (jamais public direct) : `$request->file('photo')->store('students/photos', 'tenant')`.
